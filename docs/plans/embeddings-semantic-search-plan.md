## Summary

Add local embedding generation and semantic search using **Arcana** as the RAG backbone, with a custom semantic chunker that detects topic boundaries via embedding similarity drops. Memory entries get a 5-signal scoring model (semantic similarity, FTS rank, importance, recency decay, activation strength) with GNN-inspired spreading activation. Documents get folder-based spreading activation where folders act as graph nodes ŌĆö retrieving a chunk from any doc in a folder boosts all sibling docs. Fully in-process, no external APIs, no GPU required.

Detailed technical spec: `docs/phase3-embedding-plan.md`

#### Design Approach

**Dual activation model ŌĆö GNN without GNN:**

```
Memory System                          Document System
(implicit edges via cosine sim)        (explicit edges via folder structure)

  Query embedding                        Arcana hybrid search
       |                                      |
  5-signal scoring                    folder activation_boost multiplier
  (sem + fts + importance +                   |
   recency + activation)              Retrieved chunk from folder X
       |                                      |
  Top-K retrieved                     Spread: boost folder X activation
       |                                      |
  Spread to 5 nearest neighbors       Next search: all docs in X score higher
  (bump decay_factor, cap 1.5)        (non-recursive, direct siblings only)
       |                                      |
  Daily cooling cron                   Daily cooling cron
  (decay_factor -> 1.0)               (activation_boost -> 1.0)
```

**Folder-as-graph model:**
- Folders are nodes with their own embeddings (mean of child doc embeddings)
- Folder membership = explicit graph edges (non-recursive)
- Folders are independently searchable as topic-level clusters

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Embedding model | gte-small vs all-MiniLM | **gte-small (384D)** | Best quality/size tradeoff; ~70MB; Bumblebee native |
| Memory scoring signals | Cosine-only vs multi-signal | **5-signal weighted** | Matches cognitive retrieval: semantic + lexical + importance + recency + reinforcement |
| Spreading activation edges | Explicit graph vs embedding space | **Cosine similarity as implicit edges** | No adjacency matrix needed; embedding space IS the graph |
| Document association | Embedding-only vs folder-aware | **Both: embedding + folder structure** | Folders provide explicit grouping that cosine sim can't capture |
| Folder recursion | Recursive vs flat | **Flat (non-recursive)** | Prevents activation explosion; direct siblings only |
| Folder embedding | Separate model vs aggregate | **Mean of child chunk embeddings** | No extra model calls; recomputed on ingest |
| Decay cooling | Hourly vs daily | **Daily cron** | Sufficient granularity; matches memory decay model |
| Config toggle | Compile-time vs runtime | **Runtime config flag** | `config :assistant, :embeddings, enabled: true/false` |

## Open Questions

### Resolved

- [x] **Single model vs separate**: Single gte-small shared by Arcana + memory embeddings via Nx.Serving
- [x] **Folder recursion depth**: Non-recursive (direct siblings only) to prevent activation explosion
- [x] **Folder embedding strategy**: Mean of child chunk embeddings, recomputed on each ingest
- [x] **Activation cap**: 1.5x for memories (embedding neighbors), 1.3x for folders (more conservative)
- [x] **Cooling rate**: 10% exponential cooldown per day for both systems

### Open

- [ ] **Backfill strategy**: Run mix task before or after deploy? Batch size for large memory stores?
- [ ] **Arcana version pinning**: Confirm v1.3 API stability for chunker/embedder behaviours
- [ ] **EXLA compilation**: CPU-only EXLA may need build tooling (cmake) in CI/CD
