# Implementation Plan: Identity Bridge Fix + Passive Space Context

> Generated by `/PACT:plan-mode` on 2026-03-04
> Status: APPROVED

## Summary

Fix the broken web-to-chat identity bridge that prevents GChat messages from appearing in the web workspace UI. Replace fragile heuristic linking (count all users, sole settings_user) with deterministic email-based matching between `settings_users.email` (admin-controlled, canonical) and GChat's `user["email"]` (Google Workspace, trusted). Then layer on passive Space context injection so that all users in a shared GChat Space automatically receive read-only context from each other's bot interactions — no LLM call, just appended to conversation history.

#### Design Approach

**Identity Resolution State Machine:**

```
settings_user logs in → ensure_linked_user:
  ├── user_id → real user (channel != "settings") → DONE ✓
  ├── user_id → pseudo-user (channel = "settings") → email match?
  │   ├── YES: upgrade (re-link to real user, migrate conversations) → DONE ✓
  │   └── NO:  keep pseudo-user (no chat user with this email yet)
  └── user_id is nil → email match?
      ├── YES: link to matched user → DONE ✓
      └── NO:  create pseudo-user (first-time, no chat user yet)

GChat message arrives → UserResolver.resolve:
  ├── user_identity exists → use existing user_id → DONE ✓
  └── user_identity not found → find_linked_user_id_by_email:
      ├── Match: settings_user.email matches GChat email, user_id is real → link identity to that user
      ├── Match: settings_user.email matches, but user_id is pseudo → create real user, migrate pseudo, link identity
      └── No match: create new user + identity (admin links manually later)
```

**Passive Space Context Flow:**

```
User A messages bot in Space X
  → Dispatcher processes normally (resolve → engine → reply)
  → After reply: enqueue SpaceContextFanoutWorker(space_id, sender_user_id, question, response)
  → Worker:
      1. Query user_identities for other users in Space X (WHERE left_at IS NULL)
      2. Skip DM spaces (spaceType = "DIRECT_MESSAGE")
      3. For each other user:
         a. get_or_create_perpetual_conversation(user_id)
         b. batch_append_messages with role="system", metadata.type="space_context"
      4. Rate limit: max 50 recipients, debounce rapid messages
```

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Email storage | users.email / user_identities.email / joins only | `users.email` | User-level attribute; enables direct lookups; unique partial index |
| Pseudo-user strategy | In-place upgrade / create new + migrate | Re-link + migrate conversations | Least disruptive; pseudo-user is just a placeholder |
| Space membership | New table / user_identities / GChat API | `user_identities` query | Already tracks (user, space) pairs; zero maintenance |
| Space context message role | New "context" role / system + metadata / user with prefix | `system` + `metadata.type = "space_context"` | No schema change; LLMs handle system messages correctly |
| Fan-out mechanism | Synchronous / Oban / PubSub | Oban async job | Non-blocking; retry built-in; observability |
| Space context in web UI | Hidden / visible-distinct / always shown | Visible but collapsed/dimmed | Users benefit from seeing Space activity; shouldn't dominate |
| Space removal handling | Delete identity / soft-delete with left_at | `left_at` timestamp on user_identities | Preserves history; filterable |

## Cross-Cutting Concerns

| Concern | Status | Notes |
|---------|--------|-------|
| Security | Needs attention | Space context injection means User A's messages appear in User B's conversation. Acceptable for shared spaces (inherently public). Must NOT happen for DMs. DM guard required. |
| Performance | Needs attention | Fan-out: O(N) DB writes per space message. Mitigated by Oban async, max 50 recipients, debounce. |
| Privacy | Needs attention | Space context is visible to other users by design. Admin-controlled access means all users in a space are trusted. |
| Observability | Ready | Existing telemetry in Dispatcher. Oban provides job metrics. Add Telemetry events for fan-out. |

## Open Questions

### Resolved Decisions
- [x] **Space context opt-in vs default-on**: Default ON for all shared spaces. Matches the "public space = shared context" vision.
- [x] **Space context in web UI**: Visible but dimmed/collapsed in workspace chat. Users see what context the bot has.
- [x] **Space context retention**: Archive with conversation — follows normal 30-day archival rules via ConversationArchiver.
- [x] **Compaction handling**: Include in compaction — space context gets summarized along with normal conversation history.

### Require Further Research
- [ ] **GChat email availability in production**: Verify that the v2 Workspace Add-on format includes `user["email"]` with current API scopes. If not, may need to add the `chat.users.readstate` or similar scope.
- [ ] **GChat DM space detection**: Confirm `spaceType: "DIRECT_MESSAGE"` is reliably present in webhook payload for DM filtering.
