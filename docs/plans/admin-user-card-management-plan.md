# Implementation Plan: Admin User Card Management with Shared Component

> Generated by `/PACT:plan-mode` on 2026-03-03
> Status: PENDING APPROVAL

## Summary

Refactor admin user management from tables to a card-based system using a new shared `CardGrid` component extracted from the workflow cards pattern. Each user gets a card with status badges, edit/delete icons, and a master disable toggle. Clicking edit navigates to a full user detail view for permissions management. A new `disabled_at` timestamp on `settings_users` enables admins to disable users entirely, with three-layer enforcement (login, session, LiveView mount).

#### Design Approach

**Shared CardGrid Component** (`card_grid.ex`):
- Slot-based Phoenix function component
- Accepts: `id`, `items` (list), `search_field` (assign name), `search_placeholder`, `empty_message`
- Provides `:inner_block` slot yielding each item for consumer-defined card content
- Handles: search input (`phx-change`), grid layout (`sa-workflow-grid`), empty state (`sa-empty`)
- Consumers (workflows, user cards) define card appearance via the slot

**User Detail View** — Nested within SettingsLive using `current_admin_user` assign pattern (mirrors existing `current_app` pattern in `settings_page.ex`). Route: clicking edit card sets `current_admin_user` assign, which conditionally renders `user_detail.ex` instead of the card grid. Back button clears the assign. No new LiveView or router route needed.

**Disabled Enforcement** — Three-layer defense-in-depth:
1. **Login gate**: Block new sessions for disabled users
2. **Session validation**: Return nil from `get_settings_user_by_session_token/1` if disabled
3. **LiveView mount**: Redirect with flash in `on_mount(:require_authenticated)`

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Disabled field type | Boolean vs timestamp | `disabled_at` (utc_datetime) | Audit trail, consistent with `confirmed_at` pattern |
| Session invalidation | Eager (delete tokens) vs lazy (check on access) | Lazy | Simpler, equally effective — LiveView heartbeat catches within seconds |
| Detail view routing | New LiveView vs nested assign | Nested assign | Consistent with `current_app` pattern, shares auth state |
| Card component design | Assigns-only vs slots | Slots (`inner_block`) | Maximum flexibility for different card content |
| Search implementation | Client-side JS vs server-side assign filter | Server-side (phx-change) | Consistent with `filter_active_models` pattern, works with LiveView |
| Workflow card refactor | Same PR vs separate | Same PR | Validates shared component design |
| Disabled users' LLM access | Block all vs dashboard-only | Dashboard-only | Disable controls settings access; chat/LLM is separate concern |
