## Summary

Comprehensive user management feature that makes the Add User page mirror the Edit User page, enforces access scopes that currently exist but are never checked, and introduces per-user spending limits. Three major workstreams: (1) schema enhancements + upfront user creation, (2) dual-layer access scope enforcement in both web UI and chat engine, (3) spending limits with DB-backed usage ledger and configurable hard/soft caps.

#### Design Approach

**Dual-layer access scope enforcement:**
- Layer 1 (Web UI): LiveView on_mount hook + sidebar/section visibility gating
- Layer 2 (Chat Engine): Pre-skill-execution check in SubAgent alongside existing SkillPermissions

**Spending limits — Ledger pattern:**
- Separate config table (`spending_limits`) and tracking table (`usage_records`)
- Upsert pattern for usage: one row per user per period, atomic increments
- Pre-flight check before LLM calls, post-flight recording after
- Enforcement at `LLMRouter.chat_completion/3` level (single point covers all callers)

**Upfront user creation:**
- Admin creates user -> transaction creates both allowlist entry AND settings_user
- settings_user with `hashed_password: nil` (magic link for first login)
- Existing auth flows handle nil password safely

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Budget unit | Tokens vs Dollars | Dollars (cents) | Model-agnostic; OpenRouter provides cost; tokens meaningless across models. **Cross-provider cumulative**: budget tracked across ALL providers. OpenRouter returns `cost` directly; Direct OpenAI needs `cost_for_model()` pricing lookup fallback. |
| Budget period | Daily/Weekly/Monthly | Monthly with configurable reset day | Simple, intuitive; schema supports daily/weekly later |
| Enforcement mode | Hard cap vs soft warning | Configurable per-user (admin chooses) | Flexibility; default hard_cap=true, warning at 80% |
| Scope enforcement | Mount guard vs per-section vs both | Both (mount guard + per-section visibility) | Defense in depth |
| Usage storage | JSONL analytics vs DB table | Separate DB table (usage_records) | JSONL not queryable; need indexed aggregation |
| Budget check perf | Direct DB vs ETS cache | Cached DB query (60s TTL in Engine GenServer) | Correct by default; negligible overhead |
| Pre-created users | nil password + magic link | nil hashed_password | Existing auth handles safely; magic link works |
| Full name | Single field vs first/last | Single full_name | Matches display_name pattern; no i18n issues |
| Scope->skill mapping | Hardcoded vs configurable registry | Configurable registry (like @skill_prefix_to_group) | Extensible for new domains |
| spending_limits schema | Column on settings_users vs separate table | Separate table | Flexible (multiple period types); settings_users already wide |

## Open Questions

### Resolved Decisions
- [x] **Soft vs hard limit default**: Hard cap (block at limit). Admin can configure soft warning per-user. Warning fires at 80%.
- [x] **Grace period on limit**: Complete the current call, block the NEXT one. Cost is unknown before the call; post-check is the standard approach.
- [x] **Chat scope meaning**: Removing "chat" scope disables advanced features (skill execution, tools, integrations) but user can still have basic conversation.
- [x] **Scope enforcement for admin users**: Admins bypass all scope checks (always have full access). Prevents admin self-lockout.

### Research Completed
- [x] **Direct OpenAI cost estimation**: RESOLVED. Both OpenRouter and OpenAI return `prompt_tokens` and `completion_tokens` on every call. OpenRouter also returns `cost` (dollars). Direct OpenAI returns `cost: nil` — need a per-model pricing lookup to compute cost from tokens. The `record_llm_analytics` functions in `loop_runner.ex:281` and `sub_agent.ex:1587` already capture the full usage map including model name, so the spending ledger can compute cost via `(prompt_tokens * input_price + completion_tokens * output_price)` for Direct OpenAI calls.
- [x] **Welcome email for pre-created users**: RESOLVED. Auto-send invite magic link email when admin creates user. User clicks link, lands in app, sets up profile. Uses existing `deliver_login_instructions` flow.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Race condition: admin creates + user self-registers same email | Low | Medium | Unique constraint handles it; registration finds existing record |
| Spending check performance on every LLM call | Low | Low | Cached query (60s TTL); one SUM per user per minute |
| Incomplete scope->skill mapping for new domains | Medium | Medium | Default-deny for unmapped prefixes; log denials |
| Pre-created users never claimed (orphans) | Low | Low | Admin can deactivate; add "never logged in" indicator |
| Cost field nil from some providers | Medium | Medium | Fallback to token-based estimation; track tokens alongside cost |
| Analytics JSONL vs DB usage divergence | Low | Low | Both written on each call; ledger is source of truth for spending |
