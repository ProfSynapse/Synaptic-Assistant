# Implementation Plan: Google Workspace Sync + Multi-Channel Communication

> Generated by `/PACT:plan-mode` on 2026-03-02
> Status: APPROVED

## Summary

Two feature streams: (1) A sync engine that downloads Google Docs→Markdown and Sheets→CSV into an encrypted, agent-only local workspace with granular per-drive/per-folder scoping and read/write permissions, with phased bidirectional upload, and (2) extending the existing channel adapter architecture to support Telegram, Slack, and Discord (WhatsApp ruled out due to Meta AI chatbot ban). Implementation order: channels first (lower complexity, enables sync conflict notifications), then sync download-only, then sync bidirectional for regular files.

### User Decisions (Resolved)
- **Third channel**: Discord (WhatsApp banned for AI chatbots per Meta policy Jan 2026)
- **Sync scope**: Granular — users pick specific drives AND folders, with per-folder/file read-only vs read-write permissions. No deleting — all removals go through archive (existing pattern).
- **Local file encryption**: Yes — Cloak encryption at rest. Agent workspace is airgapped (no external tool access needed). Transparent encrypt/decrypt in `Sync.FileManager`.

- [ ] Google Sheets API `spreadsheets.values.update` for CSV→Sheet (Phase 4)

#### Dependencies to Map
- `google_api_drive` (existing) — covers Drive Changes API
- `file_system` (existing) — covers local file watching (`:fs` NIF)
- `telegex` or raw `req` — Telegram Bot API client

#### Key API Findings
- **Google Docs→Markdown**: Drive API v3 `export` with `mimeType=text/markdown` — native, no Pandoc needed
- **Google Sheets→CSV**: `export` with `mimeType=text/csv` — first sheet only; multi-sheet needs Sheets API `spreadsheets.get`
- **CRITICAL**: Google Workspace files (Docs/Sheets) CANNOT be updated via `files.update`. Bidirectional upload requires Google Docs API (`batchUpdate`) and Sheets API (`spreadsheets.values.update`) — extremely complex. Consensus: download-only first.
- **Drive Changes API**: `changes.list` with `startPageToken` is efficient (returns only deltas). Push notifications (`changes.watch`) require domain verification + 7-day renewal — defer to later phase.
- **Rate limits**: Drive 20K calls/100s per user, 3 writes/sec sustained. Existing `:fuse` circuit breaker handles this.

#### Questions to Resolve
- [x] WhatsApp viability → **NOT VIABLE** — Meta policy (effective Jan 2026) bans general-purpose AI chatbots on WhatsApp Business Platform. Recommend Discord as third channel.
- [ ] Discord bot approval requirements for >100 servers (if applicable)

#### Design Approach

**Channel Layer**: Extend existing `Channels.Adapter` behaviour with optional callbacks for capabilities (typing indicators, rich messages, reactions). Extract shared `Channels.Dispatcher` from `GoogleChatController`. Per-channel webhook controllers with dedicated auth plugs. `Channels.Registry` maps atoms to modules.

**Sync Engine**: Polling via Oban cron (every 60s) using Drive Changes API `startPageToken` cursor. File-level sync granularity. Last-write-wins conflict resolution with `.conflict` copies and chat notification. Filesystem storage with Ecto metadata tracking. Per-user workspace directory isolation.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Sync strategy | A) Poll-only via Oban, B) Push notifications (Drive watch), C) Hybrid push+poll | **A — Poll-only for v1** | Push requires domain verification + 7-day renewal. Poll with Changes API is efficient (only returns deltas). Add push later. |
| Channel abstraction | A) Extend existing behaviour, B) Plugin architecture | **A — Extend behaviour** | Existing pattern works well. Finite set of channels doesn't warrant dynamic loading. |
| Conflict resolution | A) Last-write-wins, B) Three-way merge, C) User-decides | **A — LWW with detection** | Simple, predictable. Create `.conflict` copies. Notify via chat. |
| Workspace files upload | A) Bidirectional, B) Download-only, C) Phased | **C — Phased** | Download-only first (immediate value). Upload for regular files next. Workspace file upload last (requires Docs/Sheets API). |
| Third channel | A) WhatsApp, B) Discord, C) Defer | **B — Discord** ✅ | WhatsApp banned for AI chatbots (Meta policy Jan 2026). Discord has no such restriction, mature `nostrum` library, OAuth2 bot flow. User confirmed. |
| Channel credentials | A) All in oauth_tokens, B) Per-channel tables, C) Polymorphic | **C — Polymorphic `channel_connections`** with encrypted credentials blob + `oauth_tokens` for Slack (standard OAuth2) |
| User identity | A) Separate per channel (v1), B) Cross-channel linking | **A for v1** | Include `user_identities` table for future linking but don't auto-link. Explicit opt-in only (security requirement). |
| Sync trigger | A) Per-user, B) System-wide | **A — Per-user** | Users control what syncs. Respects Drive scoping. |

### 🛡️ Security Concerns
**Effort**: Medium-High

#### Pre-Launch Blockers (CRITICAL)
1. **Telegram webhook is unauthenticated** — existing route at `POST /webhooks/telegram` returns 200 with no verification. Must add `TelegramAuth` plug before going live.
2. **Path traversal in sync** — Google Drive filenames are user-controlled. `Sync.FileManager` MUST validate all paths (reject `../`, absolute paths, symlinks). Extend `WorkflowWorker.resolve_path/1` pattern.

#### Webhook Security Checklist (Per Channel)
- [ ] Dedicated auth plug with signature/token verification
- [ ] Timestamp validation to prevent replay attacks
- [ ] Constant-time comparison via `Plug.Crypto.secure_compare/2`
- [ ] Request body size limit
- [ ] Rate limiting per source IP and per user
- [ ] Input validation on all message fields
- [ ] Error responses that don't leak internals

#### Data Protection
- Per-user workspace directory isolation: `{sync_root}/{user_id}/`
- Filename sanitization: allowlist pattern `~r/^[a-zA-Z0-9._\- ]+$/`
- Channel credentials encrypted via `Cloak` (`Encrypted.Binary`)
- Cross-channel identity linking: **explicit opt-in only** (no auto-linking by email)
- Sentinel gate should consider originating channel's trust level

## Database Schema Summary

### New Tables (6)
1. **`synced_files`** — Maps Google Drive files to local encrypted copies (user_id, drive_file_id, local_path, sync_status, checksums, timestamps). Content encrypted via Cloak before writing to disk.
2. **`sync_cursors`** — Per-user per-drive Changes API cursor (startPageToken)
3. **`sync_history`** — Append-only audit log of sync operations
4. **`sync_scopes`** — Granular per-user sync permissions (user_id, drive_id, folder_id, access_level: `read_only` | `read_write`). Users select specific drives AND folders. Controls what the agent can read vs write. No delete operations — all removals go through archive.
5. **`channel_connections`** — Per-user channel configs with encrypted credentials blob (polymorphic: telegram, slack, discord)
6. **`user_identities`** — Cross-channel identity mapping (for future linking, explicit opt-in only)

### Modified Tables (1)
- **`oauth_tokens`** — Expand provider CHECK constraint to include `'slack'`

### Migration Sequence
```
1. create_sync_cursors        (FK → users)
2. create_synced_files         (FK → users)
3. create_sync_history         (FK → synced_files, cascading delete)
4. create_channel_connections  (FK → users)
5. create_user_identities      (FK → users)
6. expand_oauth_tokens_provider_constraint
```

## Specialist Conflict Resolution

| Conflict | Resolution | Rationale |
|----------|------------|-----------|
| Push+poll (architect) vs poll-only (backend, preparer) | **Poll-only for v1** | Push requires domain verification + renewal. Poll with Changes API is efficient. Add push as optimization later. |
| `channel_configs` system-level (architect) vs `channel_connections` per-user (database) | **Both needed** | System config (bot token, webhook URL) goes in env vars/runtime config. Per-user config (which channels connected, user-specific tokens) goes in `channel_connections` table. |
| Separate identities (architect) vs `user_identities` table (database) | **Include table, don't auto-link** | Create `user_identities` schema for future use. V1: separate identities per channel. Cross-channel linking is explicit opt-in only (security requirement). |
