# Implementation Plan: requires_approval Gate for Dangerous Skills

> Generated by `/PACT:plan-mode` on 2026-03-05
> Status: IMPLEMENTED

## Summary

Add a deterministic `requires_approval` gate to the sub-agent execution pipeline that intercepts dangerous skill calls (email.send, calendar.create, calendar.update, files.archive, workflow.create, workflow.run) before execution, pauses the sub-agent, and surfaces the action to the user through the orchestrator for explicit approval. Reuses the existing `request_help` pause/resume mechanism with zero new GenServer message types.

#### Design Approach

**Interceptor pattern** in `execute_use_skill`, composing with the existing pipeline:

```
execute_use_skill pipeline:
  1. SkillPermissions.enabled? check (admin policy)
  2. Scope check (skill in agent's allowed list)
  3. Sentinel check (LLM-based alignment)
  4. APPROVAL GATE (deterministic, field-based)  <-- NEW
  5. execute_skill_call (actual execution)
```

**Pause mechanism**: Reuse existing `{:loop_paused, reason, partial_history, synthetic_tc}` message to the GenServer. The GenServer's existing `handle_info` at line 384 transitions to `:awaiting_orchestrator` — no new GenServer clauses needed.

**Synthetic TC**: Build `%{id: "approval_#{unique_id}"}` to satisfy the `:loop_paused` tuple shape. The GenServer stores it in `pending_help_tc` but the resume path (`handle_call({:resume, update})`) just sends `{:resume, update}` to the loop task pid regardless of TC content.

**Resume handling**: A NEW `receive` block inside `execute_use_skill` (separate from the existing `request_help` receive block at line 652). Only one is ever active at a time — either request_help OR approval gate, never both simultaneously.

**Resume decision parsing** (structured via `send_agent_update` tool):
- The `send_agent_update` tool gains a new optional `approved: boolean` field
- `approved: true` -> execute the skill call
- `approved: false` + `message` -> return the message as feedback to sub-agent LLM (it adjusts and retries `use_skill`, re-triggering the gate)
- `approved: false` without message -> return generic cancellation to sub-agent LLM

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Gate location | Inline in `execute_use_skill` vs separate module | Inline | ~30 lines of code; extract later if it grows beyond ~50 lines |
| Pause mechanism | New GenServer message vs reuse `{:loop_paused, ...}` | Reuse existing | Zero new GenServer code; same `:awaiting_orchestrator` state |
| Resume disambiguation | String prefix parsing vs structured boolean field | `approved: boolean` field on `send_agent_update` | Deterministic; no string sniffing; LLM fills a boolean via tool schema |
| Feedback loop | Keep paused and iterate vs return to LLM loop | Return to LLM loop | Sub-agent LLM adjusts parameters, calls `use_skill` again, gate re-triggers naturally |
| Registry lookup timing | Double lookup vs move earlier | Move `Registry.lookup` earlier in `execute_use_skill` | Avoid redundant lookup; pass `skill_def` to `execute_skill_call` |

#### Interface Contracts

**Approval reason format** (sub-agent -> orchestrator via `awaiting_reason`):
```
[APPROVAL_REQUIRED] Skill "email.send" requires user approval.

Proposed action:
  to: bob@example.com
  subject: Meeting follow-up
  body: Hi Bob, here are the action items from our meeting...
  cc: (none)
```

The orchestrator presents this to the user in plain language. The user replies naturally — the orchestrator interprets the response and uses `send_agent_update` with `approved: true/false` (+ optional `message` for feedback).

**Resume format** (orchestrator -> sub-agent via `send_agent_update`):
- Approval: `send_agent_update(agent_id: "...", approved: true)`
- Cancellation: `send_agent_update(agent_id: "...", approved: false)`
- Feedback: `send_agent_update(agent_id: "...", approved: false, message: "User wants to change the subject to 'Action Items' and add CC alice@example.com")`

**`send_agent_update` schema addition:**
```json
"approved": {
  "type": "boolean",
  "description": "For approval-gated skills only. true = user approves, proceed with execution. false = user denies or wants changes (include feedback in message)."
}
```

#### Implementation Detail: Sub-agent Gate

The gate intercepts in `execute_use_skill` (currently lines 748-806). After sentinel approval (line 793-794), before `execute_skill_call`:

```
# Pseudocode — actual implementation by coder
def execute_use_skill(tc, args, dispatch_params, engine_state, genserver_pid) do
  skill_name = args["skill"]
  skill_args = sanitize_args(args["arguments"])

  # ... existing scope + permission checks ...

  case Sentinel.check(...) do
    {:ok, :approved} ->
      # Move Registry.lookup earlier (currently inside execute_skill_call)
      case Registry.lookup(skill_name) do
        {:ok, %{requires_approval: true} = skill_def} ->
          # Build structured approval reason from skill_def.parameters + skill_args
          reason = build_approval_reason(skill_name, skill_args, skill_def)
          synthetic_tc = %{id: "approval_#{System.unique_integer([:positive])}"}

          # Pause using existing mechanism
          send(genserver_pid, {:loop_paused, reason, nil, synthetic_tc})

          # Block until orchestrator resumes via send_agent_update(approved: bool)
          receive do
            {:resume, %{approved: true}} ->
              execute_skill_call(tc, skill_name, skill_args, skill_def, ...)

            {:resume, %{approved: false, message: feedback}} when is_binary(feedback) ->
              {tc, "APPROVAL_DENIED: User requested changes: #{feedback}\n\n" <>
                   "Adjust your approach based on the feedback and try again."}

            {:resume, %{approved: false}} ->
              {tc, "APPROVAL_DENIED: User cancelled this action."}
          after
            300_000 -> {tc, "Approval request timed out (5 minutes)."}
          end

        {:ok, skill_def} ->
          execute_skill_call(tc, skill_name, skill_args, skill_def, ...)

        {:error, :not_found} ->
          {tc, "Skill not found: #{skill_name}"}
      end

    {:ok, {:rejected, reason}} -> ...
  end
end
```

#### Implementation Detail: Orchestrator Prompt Addition

Add to `orchestrator.yaml` after the existing Rules section:

```yaml
## Approval Handling

When get_agent_results shows an agent status of "awaiting_orchestrator" with a reason
starting with "[APPROVAL_REQUIRED]", the agent needs user approval before proceeding:

1. Present the action details to the user clearly and concisely
2. Ask for approval: proceed, modify, or cancel
3. Based on user response:
   - If user approves: send_agent_update with approved=true
   - If user cancels: send_agent_update with approved=false
   - If user wants changes: send_agent_update with approved=false and message with feedback
     (the agent will adjust and request approval again)

IMPORTANT: Always present the FULL action details from the approval request. The user
must see exactly what will happen (recipient, subject, body, etc.) before approving.
```
