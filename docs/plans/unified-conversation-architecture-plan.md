## Summary

Build a unified cross-channel conversation architecture where each user has ONE perpetual conversation that never closes. Messages from any channel (Telegram, Slack, Discord, Google Chat) flow into a single thread. The bot knows which channel a message came from, responds on that channel, and can proactively message other channels. Admin users can view all users' memories from the settings dashboard.

The implementation bridges the current "two-world problem" (ephemeral string conversation IDs in Engine vs UUID-based DB tables) by adding: a `user_identities` join table for cross-channel identity, a `UserResolver` module for auto-creating users on first contact, Engine re-registration by user UUID, batch message persistence at turn end, and a `ReplyRouter` for channel-agnostic outbound messaging.

#### Design Approach

**New Dispatch Flow:**
```
Webhook → Controller → Dispatcher.dispatch(adapter, message)
    │
    ├── UserResolver.resolve(channel, external_id, metadata)
    │     → {:ok, %{user_id: uuid, conversation_id: uuid}}
    │
    ├── ensure_engine_started(user_id, conversation_id, message)
    │     Engine registered by user_id in EngineRegistry
    │
    ├── Engine.send_message(user_id, content)
    │     → {:ok, response_text}
    │     → (async) batch persist turn messages to DB
    │
    └── ReplyRouter.reply(origin, response_text, opts)
          → :ok
```

**Module Boundaries:**

| Module | Responsibility | Data Ownership |
|--------|---------------|----------------|
| `UserResolver` | Platform identity → DB user + conversation | Writes: users, user_identities, conversations |
| `ReplyRouter` | Outbound message routing to channels | Reads: user_identities, Channels.Registry |
| `Dispatcher` | Orchestrate inbound flow (thin coordinator) | No data — coordinates UserResolver, Engine, ReplyRouter |
| `Engine` | LLM loop, sub-agent dispatch, in-memory history | In-memory messages; delegates persistence to Store |
| `Store` | DB CRUD for conversations, messages | All DB writes for conversation/message domain |

#### Key Decisions

| Decision | Resolution | Rationale |
|----------|-----------|-----------|
| Engine registration key | `user_id` UUID (not conversation_id) | 1:1 user↔conversation mapping; user_id available immediately from UserResolver; simpler lookup |
| conversations.channel for perpetual | Set to `"unified"` | Explicit, searchable, avoids NULL semantics; sub-agent conversations keep their channel value |
| users.external_id/channel columns | Keep NOT NULL, drop UNIQUE constraint | Backward-compatible; columns become "primary identity" (first channel); user_identities is authoritative |
| Cross-channel linking trigger | Admin-only for v1 | Eliminates identity spoofing risk entirely; can add user self-service later |
| Message persistence mechanism | Async via Task.Supervisor at turn end | Lightweight; doesn't block reply; Engine memory serves as write buffer |
| Message persistence scope | Batch all turn messages (user + tool_calls + tool_results + assistant) | One DB transaction per turn via Ecto.Multi |
| Admin memory visibility | is_admin check in query layer (not RLS) + audit logging | Consistent with existing codebase pattern; Logger.info on cross-user access |
| User auto-creation | Allowlist required; reject unknown platform users | Prevents spam; admin manages allowed users via settings dashboard |
| Data retention | Compaction for Engine context; full transcript searchable in DB | No hard deletion; messages table is permanent record |
| UserResolver race condition | Upsert with ON CONFLICT on user_identities unique constraint | Standard Ecto pattern; retry lookup on conflict |

### 🛡️ Security Controls
**Effort**: Medium (woven into implementation)

#### Threat Vectors & Mitigations

| Threat | Mitigation | Owner |
|--------|-----------|-------|
| Cross-channel identity spoofing | Admin-only linking for v1; no automatic cross-channel association | UserResolver |
| Platform ID injection | Regex validation per channel before DB insert (Telegram: `^\d+$`, Slack: `^[A-Z0-9]+$`, etc.) | UserResolver |
| Unauthorized user access | Allowlist required — UserResolver rejects unknown platform users; no auto-creation without allowlist entry | UserResolver |
| Unauthorized memory access | is_admin check on all cross-user queries; non-admins scoped to own user_id | Store query layer |
| ReplyRouter sending to wrong user | Always resolve target through user_identities table; never accept arbitrary space_id | ReplyRouter |
| Unbounded data growth | CASCADE DELETE: user → conversation → messages → memory_entries; design deletion path upfront | Schema constraints |

#### Input Validation Requirements

```elixir
# Platform ID format validation (in UserResolver)
@platform_id_patterns %{
  telegram: ~r/^\d{1,20}$/,
  slack: ~r/^[A-Z0-9]{2,20}$/,
  discord: ~r/^\d{1,20}$/,
  google_chat: ~r/^users\/\d{1,20}$/
}
```

## Open Questions

### Resolved by User

- [x] **User allowlist for auto-creation**: **Allowlist required.** Platform users must be on an allowlist to auto-create DB users. Unknown users who message the bot are rejected. This prevents spam user creation and limits access to authorized users. Implementation: UserResolver checks allowlist before creating user; unauthorized messages get a configurable rejection response.
- [x] **Data retention policy**: **Compaction + full transcript searchable.** Messages are compacted (summarized) for the Engine's working context, but ALL messages remain in the DB and are searchable. No hard deletion or archival limit. The `messages` table is the full transcript; the `conversations.summary` is the compacted working memory.
- [x] **Admin memory access audit logging**: **Yes, log admin access.** Add a Logger.info entry when an admin views another user's memories/conversations. Lightweight audit trail without a separate audit table.

### Require Further Research
- [ ] **Engine idle timeout**: With perpetual conversations, should the Engine GenServer have an idle timeout (e.g., 30 minutes)? On restart, it hydrates from DB. This saves memory but adds cold-start latency.
- [ ] **Sub-agent DB conversation creation**: Currently sub-agents don't create DB conversations. Should they start doing so for FK integrity, or is the parent's perpetual conversation sufficient?
- [ ] **Allowlist implementation**: Where does the allowlist live? Options: (a) new `allowed_users` table with (channel, external_id), (b) reuse existing `settings_user_allowlist` pattern, (c) admin manages via settings UI. Need to decide schema and management UX.
