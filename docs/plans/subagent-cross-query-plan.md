# Implementation Plan: Non-Interrupting Subagent Cross-Query

> Generated by `/PACT:plan-mode` on 2026-03-06
> Status: IN PROGRESS

## Summary

Add a read-only cross-agent query path so:
- parallel subagents know the mission of sibling subagents
- a subagent can query another subagent without interrupting it
- the orchestrator can ask a running subagent a question without resuming or pausing it

Do this by reusing the existing orchestrator, subagent, and LLM routing architecture.

The key enabling change is that subagents must update an inspectable transcript snapshot as they run, not only at pause/completion.

#### Product Direction (from request)
- Do not build a separate coordination architecture if the current one can support the behavior.
- Parallel subagents should know the mission of sibling subagents.
- A subagent should have a `query_subagent` capability that reads another agent's transcript snapshot and answers via its own LLM summarization call.
- Querying another subagent should not interrupt the target subagent.
- When the orchestrator asks a subagent a question, that should also be non-interrupting and should use a branch/snapshot question-answer flow rather than `resume/2`.
- A subagent may still use `request_help` to ask the orchestrator to relay something to another subagent when needed.

#### Design Constraints
- Preserve `request_help` and `send_agent_update` as the true interrupt/resume path.
- Avoid adding a second long-lived agent runtime.
- Reuse the `:sub_agent` model role for snapshot-question answering when possible.
- Add a real orchestrator kill switch for subagents:
  - immediate termination
  - transcript truncates at the last known point
  - explicit cancelled/killed terminal state
- Keep transcript sharing bounded and sanitized:
  - strip system prompt text
  - share only user/assistant/tool/tool_call history
  - cap size to avoid runaway prompt growth

### 🏗️ Architecture Phase
**Effort**: High

#### Target Behavior

```text
Orchestrator dispatches parallel wave
  -> each subagent gets sibling mission roster
  -> each subagent updates live transcript snapshot while running

Subagent A wants context from Subagent B
  -> A calls query_subagent
  -> system reads B's current snapshot
  -> system runs a separate LLM summarization/Q&A call with subagent model
  -> A receives structured answer
  -> B continues running uninterrupted

Orchestrator wants to ask Subagent B a question
  -> orchestrator calls query-running-agent tool
  -> same shared snapshot Q&A helper runs
  -> orchestrator gets answer
  -> B continues running uninterrupted

Subagent A is blocked and needs relay
  -> A calls request_help with relay intent
  -> orchestrator decides how to coordinate
```

#### Core Architectural Decision

Do not make live subagents message each other directly.

Instead:
- the target subagent remains a single running process
- cross-agent questions operate on a point-in-time transcript snapshot
- a separate short LLM call answers the question over that snapshot

This preserves current execution semantics and avoids introducing shared mutable state or peer-to-peer live control paths.

#### New/Reused Primitives

1. **Peer Mission Roster**
   - Built when the Engine schedules a parallel wave.
   - Injected into each subagent prompt/context.
   - Same-wave peers also unlock an internal coordination skill, `agents.query_subagent`, through the existing subagent skill path rather than a new runtime surface.
   - Contains:
     - `agent_id`
     - mission summary
     - granted skills
     - dependency notes if relevant

2. **Live Transcript Snapshot**
   - Stored in the `SubAgent` GenServer state.
   - Updated after each LLM response and after each tool-result batch.
   - Separate from final persistence.
   - Bounded to a recent window plus compact metadata.

3. **Shared Snapshot Query Helper**
   - Proposed module: `Assistant.Orchestrator.SubAgentQuery`
   - Inputs:
     - target agent metadata
     - target transcript snapshot
     - asker's question
     - root conversation metadata
   - Output schema:
     - `summary`
     - `answer`
     - `progress`
     - `blockers`
     - `open_questions`

4. **Two Thin Entry Points Over the Same Helper**
   - Subagent path:
     - `query_subagent` exposed as a normal skill
   - Orchestrator path:
     - a new orchestrator tool wrapper for non-interrupting subagent questioning

5. **Engine-Managed Async Dispatch State**
   - The Engine keeps per-batch dispatch/wave state instead of blocking through a full scheduler run.
   - Wave 0 launches immediately.
   - Later waves remain queued until upstream results become terminal.
   - The Engine refreshes running agent state between loop iterations and on explicit waits.

#### State Model Changes

Extend `SubAgent` state with something like:

```text
live_snapshot: %{
  status: :running | :awaiting_orchestrator | :completed | :failed,
  mission: "...",
  peer_agents: [...],
  messages: [...bounded transcript...],
  updated_at: ...,
  tool_calls_used: n,
  last_result: "...optional..."
}
```

Important distinction:
- `messages` on final result remains the completion transcript payload
- `live_snapshot.messages` is the queryable in-flight view

#### Interrupt/Stop Semantics

Current behavior today:
- `request_help` is interrupting
- `send_agent_update` resumes only paused agents
- `interrupt/1` is graceful stop, not hard kill

Target behavior:
- keep `interrupt/1` as an internal graceful mechanism where useful
- add a separate orchestrator-controlled hard stop path
- hard stop should terminate immediately and truncate the transcript at the last published snapshot
- stopped agents should end in an explicit terminal state such as `:cancelled`
- a cancelled agent is not resumed in place; continuation happens by launching a new agent seeded from the last snapshot
- cancellation should carry reason metadata such as:
  - `:user_cancelled`
  - `:orchestrator_cancelled`
  - `:off_rails`
  - `:superseded`

New query behavior should explicitly not:
- change target status
- inject new messages into the target agent's transcript
- wake a paused agent
- resume a blocked agent

#### Implementation Sequence
1. Add live in-memory transcript snapshot support to `SubAgent`.
2. Update the subagent loop to publish snapshot updates after each meaningful step.
3. Add immediate subagent termination support with explicit cancelled state and truncated transcript handling.
4. Expose termination as an orchestrator tool such as `cancel_agent`.
5. Refactor Engine dispatch to be non-blocking within the turn loop, keeping batch/wave/dependency state in Engine.
6. Add peer mission roster injection for same-wave parallel agents.
7. Build the shared `SubAgentQuery` helper using `LLMRouter` and strict structured output.
8. Expose the helper as a subagent skill: `query_subagent`.
9. Expose the same helper as an orchestrator tool for non-interrupting questioning.
10. Optionally enrich `request_help` with relay-target metadata for orchestrator-mediated coordination.
11. Add relaunch semantics so the orchestrator can start a fresh agent from a cancelled agent's last snapshot when it wants to continue work.
12. Add unit and integration coverage for both steady-state and edge cases.

#### Query Helper Prompt/Input Shape

The helper should operate on a reduced input package such as:

```text
Target agent id: research_agent
Target mission: Find the latest customer renewal blockers
Target status: running
Peer context: [...]
Transcript snapshot:
- [user] ...
- [assistant] ...
- [tool_call] ...
- [tool] ...

Question:
What has this agent already learned about blockers, and is it waiting on anything?
```

It should use the `:sub_agent` role model selection path and return strict JSON.

#### Relay via request_help

Keep relay lightweight.

Possible extension:
- `reason` remains required
- optional fields:
  - `target_agent_ids`
  - `relay_message`
  - `relay_intent`

This allows a blocked agent to ask the orchestrator to coordinate with another agent without creating direct agent-to-agent mutation paths.

## Verification Plan

### Unit Tests
- `SubAgent` publishes updated live snapshots while `:running`.
- Snapshot serialization excludes system prompt content.
- `SubAgentQuery` returns structured output and handles empty/stale snapshots safely.
- Orchestrator query tool and `query_subagent` skill both route to the same helper.

### Integration Tests
- Parallel-dispatched subagents can see sibling missions in their prompt/context.
- A running subagent can query a sibling without changing the sibling's status.
- The orchestrator can query a running subagent without interrupting it.
- A paused `:awaiting_orchestrator` agent is not resumed by snapshot queries.
- `interrupt/1` remains graceful and does not hard-kill mid-tool-call.
- `cancel_agent` terminates a running subagent immediately and returns truncated transcript state.
- User-driven cancellation propagates to the targeted subagent without waiting for another loop step.
- `request_help` relay metadata can be surfaced to the orchestrator without breaking current flows.

### Acceptance Criteria
- No new long-lived runtime or peer-to-peer messaging subsystem is introduced.
- Cross-agent queries are read-only against snapshots.
- Running subagents maintain current queryable transcript state.
- Existing interrupt/resume semantics continue to work unchanged.
- The orchestrator can immediately kill a subagent and observe a distinct cancelled terminal state.
- The orchestrator can continue cancelled work only by starting a new agent from the saved snapshot, not by reviving the killed process.

## Risks

- Live snapshot updates can increase process memory if transcripts are not bounded.
- Querying stale snapshots may produce misleading answers if update cadence is too sparse.
- Over-sharing peer context can bloat prompts and reduce subagent focus.
- If `query_subagent` is too easy to call, agents may over-coordinate and waste budget.
- Immediate termination can leave partial side effects if a skill already began an external action.

Mitigations:
- bound snapshot size
- include `updated_at` and status in query input/output
- keep peer roster compact
- document when to use `query_subagent` vs `request_help`
- clearly distinguish graceful interrupt from immediate cancellation
- surface cancellation reason and truncation status in agent results
