# Implementation Plan: Policy Presets + Allow/Ask/Deny Runtime Controls

> Generated by `/PACT:plan-mode` on 2026-03-17
> Status: IN PROGRESS

## Summary

Add an explicit runtime policy layer that resolves attempted actions to `allow`, `ask`, or `deny` without replacing the existing hard gates. The current system already has:
- access-scope enforcement
- skill enable/disable gates
- semantic alignment checks via `Sentinel`
- human approval for selected skills

The missing capability is a user/admin-directed control plane over those behaviors.

This plan adds:
- workspace policy presets: `permissive`, `default`, `strict`
- explicit policy rules at system, workspace, and user scope
- persisted approval records for inbox/history and "always allow/deny" flows
- deterministic policy resolution in sub-agent, memory-agent, and web-fetch execution paths
- settings/admin surfaces that preserve the chat-first product shape

The core design rule is:

```text
hard capability gate first
then deterministic policy
then sentinel
then approval gate only if policy says ask
```

## Specialist Perspectives

### Preparation Phase
**Effort**: Medium

#### Current State (Confirmed)
- `Assistant.AccessScopes.Enforcer` is a hard scope gate over skill domains.
- `Assistant.SkillPermissions` already combines workspace, global, user, and connector enablement into a hard allow/deny decision.
- `Assistant.Orchestrator.Sentinel` is a semantic safety layer, not a user-facing policy system.
- `Assistant.Orchestrator.ApprovalGate` pauses sub-agents when a skill is marked `requires_approval: true`.
- `Assistant.Integrations.Web.UrlPolicy` and `Assistant.Integrations.Web.HttpFetcher` already provide deterministic SSRF/network safety for web fetches.
- Personal skill toggles already exist in Settings; they are boolean capability overrides, not `allow/ask/deny` behavior controls.

#### Product Direction (from request)
- Admins need an easy top-level choice: `Permissive`, `Default`, or `Strict`.
- Admin policy should be able to force `allow`, `ask`, or `deny` for risky actions like `email.send`.
- Users may later have personal defaults/overrides, but admin policy should remain authoritative where configured.
- Chat should stay chat-first; policy editing belongs in Settings/Admin surfaces.

- Existing control surfaces:
  - `lib/assistant_web/live/settings_live/*`
  - `lib/assistant_web/components/settings_page/*`

### Architecture Phase
**Effort**: High

#### Core Decision

Policy is a new **runtime decision layer**, not a replacement for the existing safety model.

Effective flow:

```text
1. AccessScopes.Enforcer / connector gates / SkillPermissions
2. Policy resolver => allow | ask | deny
3. Sentinel semantic check
4. ApprovalGate only when policy == ask
5. Execute action
6. Preserve runtime-local safety checks in integrations
```

#### New Concepts

1. **Policy preset**
   A workspace-wide default posture:
   - `permissive`
   - `default`
   - `strict`

2. **Policy rule**
   A deterministic rule matching an action descriptor and yielding:
   - `allow`
   - `ask`
   - `deny`

3. **Policy approval record**
   A persisted record of a one-time or durable decision:
   - approve once
   - deny once
   - always allow like this
   - always ask
   - always deny

4. **Action descriptor**
   A normalized map describing what is about to happen.

#### Policy Model

```text
policy_rules
- id
- scope_type          # system | workspace | user
- scope_id            # nullable for system
- effect              # allow | ask | deny
- resource_type       # skill_call | web_fetch | connector_request
- matchers            # JSONB
- priority            # int
- enabled             # boolean
- reason              # text
- source              # system_default | preset_seed | admin | user | approval_persisted
- inserted_at / updated_at
```

```text
policy_approvals
- id
- user_id
- workspace_id or settings_user_id (depending on current ownership model)
- rule_id (nullable)
- resource_type
- request_fingerprint
- effect              # allow | ask | deny
- expires_at          # nullable
- metadata            # JSONB
- inserted_at / updated_at
```

#### Action Descriptor Shape

Examples:

```elixir
%{
  resource_type: :skill_call,
  skill: "email.send",
  domain: "email",
  action_class: :external_write,
  integration_group: "google_workspace",
  user_id: "...",
  arguments: %{"to" => "...", "subject" => "..."}
}
```

```elixir
%{
  resource_type: :web_fetch,
  host: "example.com",
  scheme: "https",
  path: "/docs",
  action_class: :read,
  user_id: "..."
}
```

#### Preset Design

The presets should seed baseline rules, not hardcode behavior in Elixir.

| Preset | Goal | Typical behavior |
|--------|------|------------------|
| `permissive` | Minimize friction for trusted/internal use | Reads `allow`, common external writes `allow`, destructive actions `ask`, blocked destinations `deny` |
| `default` | Balanced safety/productivity | Reads `allow`, external writes `ask`, destructive actions `ask` or `deny` by category, unknown hosts `ask` |
| `strict` | High-control environments | Reads `allow`, most external writes `ask`, destructive actions `deny`, unknown hosts `deny` unless allowlisted |

#### Route Placement And Why

New policy routes should live in the existing authenticated settings surface:

- `scope "/", AssistantWeb`
- `pipe_through [:browser, :require_authenticated_settings_user]`
- existing `live_session :require_authenticated_settings_user`

Why:
- policy management is authenticated, user-specific, and needs `@current_scope`
- approvals inbox must reflect the logged-in settings user
- admin policy should stay inside the existing admin/settings experience
- this avoids `current_scope` regressions and duplicate `live_session` names

#### UI Information Architecture

```text
Workspace
└── approval-needed indicator only

Settings
├── Apps & Connections
│   └── Personal per-app policy controls
├── Approvals
│   └── pending + recent decisions
└── Admin
    └── Policies
        ├── preset selector
        ├── global category defaults
        └── explicit exceptions
```

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Replace `SkillPermissions` vs layer on top | Replace / layer | Layer on top | Capability gating and runtime policy solve different problems |
| Presets in code vs seeded rules | Code branching / seeded rules | Seeded rules | Easier to inspect, override, and audit |
| Policy location for chat UI | Full editor / minimal prompt | Minimal prompt | Preserve chat-first UX |
| `requires_approval` future | Remove now / compatibility shim | Compatibility shim first | Safe migration path |
| User overrides over admin deny | Allow / restrict | Restrict by default | Admin posture should remain authoritative for risky actions |

#### Implementation Sequence

2. Add `Policy.Action` + `Policy.Resolver`
3. Wire resolver into `sub_agent.ex`
4. Wire resolver into `memory/agent.ex`

#### Implementation Detail: Resolver Order

Pseudocode:

```text
if hard gate denies -> deny
descriptor = Policy.Action.build(...)
effect = Policy.Resolver.resolve(descriptor, actor)
if effect == deny -> deny
if Sentinel rejects -> deny
if effect == ask -> ApprovalGate.await(...)
if effect == allow -> execute
```

#### Implementation Detail: Preset Seeding

Each preset should be applied through `Assistant.Policy.apply_preset/2`:
- clear prior preset-seeded baseline rules
- insert the new preset baseline
- keep admin-authored exceptions

This avoids a hidden codepath and keeps effective policy inspectable.

#### Implementation Detail: Approval Inbox

The inbox requires persistence because current approval pauses are effectively in-memory.

Each pending approval record should include:
- action summary
- originating agent id / conversation id
- skill or host being acted on
- proposed arguments or destination
- available actions:
  - approve once
  - deny once
  - always allow like this
  - always ask
  - always deny
