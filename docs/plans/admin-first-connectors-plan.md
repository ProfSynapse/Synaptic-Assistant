# Implementation Plan: Admin-First Connector Config + Card Settings Personal Overrides

> Generated by `/PACT:plan-mode` on 2026-03-05
> Status: IN PROGRESS (worktree implementation active)

## Summary

Keep the current Apps card/grid design, move all workspace credentials to Admin, and shift user behavior to:
- card-level on/off toggle for connectors
- settings icon on cards for setup/details
- personal skill/tool overrides inside card settings (for all cards)

The separate `Skill Permissions` tab is removed.

Telegram remains the personal setup exception: if user toggles ON before setup, route them to Telegram settings flow.

#### Ownership Model

#### New Domain Concepts

1. **Per-user connector toggle state** (from prior plan)
```text
settings_user_connector_states
- settings_user_id
- integration_group
- status (connected/disconnected/...)
- metadata
UNIQUE(settings_user_id, integration_group)
```

2. **Per-user skill override state** (new)
```text
user_skill_overrides
- user_id (fk users.id)
- skill_name (string)
- enabled (boolean)
- inserted_at / updated_at
UNIQUE(user_id, skill_name)
```

Effective skill availability:
```text
effective_enabled = global_enabled(skill_name) AND user_override_enabled(user_id, skill_name)
```

This keeps safety while enabling personal opt-out (like disabling `email.send` for yourself).

For HubSpot specifically, account for the newly added `hubspot.*` inventory by applying the same effective gate to every HubSpot skill name.
Recommended connector-aware extension:
```text
effective_enabled = global_enabled(skill_name)
                 AND user_override_enabled(user_id, skill_name)
                 AND connector_enabled_for_user(user_id, integration_group_for_skill(skill_name))
```
where `integration_group_for_skill("hubspot.*") == "hubspot"`.

#### Routing + Auth Scope Placement (explicit)

All work remains in:
- `scope "/", AssistantWeb` with `pipe_through [:browser, :require_authenticated_settings_user]`
- `live_session :require_authenticated_settings_user`

Why:
- Apps card toggles, card settings, and admin integrations all require `@current_scope`.
- Avoids `current_scope` assignment regressions.
- No new `live_session` is needed.
